\chapter{R-\/\+Type Game Engine}
\hypertarget{index}{}\label{index}\index{R-\/Type Game Engine@{R-\/Type Game Engine}}
\hypertarget{index_autotoc_md0}{}\doxysection{\texorpdfstring{Documentation de la Game Engine dans le Projet R-\/\+Type}{Documentation de la Game Engine dans le Projet R-Type}}\label{index_autotoc_md0}
Cette documentation se concentre sur la game engine du projet R-\/\+Type, en particulier sur l\textquotesingle{}Entity Component System (\doxylink{classECS}{ECS}), les systèmes (systems), et leur utilisation dans le fichier {\ttfamily main.\+cpp} du client. L\textquotesingle{}objectif est de clarifier les relations entre ces éléments, en expliquant comment l\textquotesingle{}\doxylink{classECS}{ECS} gère les entités, les composants et les systèmes, et comment tout cela est orchestré pour créer un jeu de type shoot\textquotesingle{}em up.

La game engine est implémentée dans les répertoires {\ttfamily include/engine} et {\ttfamily src/engine}, avec une séparation claire entre le core (\doxylink{classECS}{ECS}), les factories (pour créer des entités), et les systems (pour la logique du jeu). Le server gère le réseau \doxylink{classUDP}{UDP}, mais la game engine est principalement utilisée côté client pour la logique de jeu, le rendering et les inputs.\hypertarget{index_autotoc_md1}{}\doxysubsection{\texorpdfstring{1. Introduction à la Game Engine}{1. Introduction à la Game Engine}}\label{index_autotoc_md1}
La game engine est basée sur un paradigme \doxylink{classECS}{ECS} (Entity-\/\+Component-\/\+System), qui est une architecture courante en développement de jeux pour favoriser la composition plutôt que l\textquotesingle{}héritage. Cela permet une flexibilité accrue \+: les entités sont des IDs simples, les composants stockent les données, et les systèmes appliquent la logique.


\begin{DoxyItemize}
\item {\bfseries{Entity}} \+: Un simple ID ({\ttfamily uint32\+\_\+t}) représentant un objet dans le jeu (ex. \+: joueur, ennemi, projectile).
\item {\bfseries{Component}} \+: Des données attachées à une entité (ex. \+: Position, Velocity, Health).
\item {\bfseries{System}} \+: Des classes qui itèrent sur les entités ayant certains composants et appliquent des mises à jour (ex. \+: mouvement, collisions, rendering).
\end{DoxyItemize}

L\textquotesingle{}\doxylink{classECS}{ECS} est défini dans {\ttfamily ecs.\+hpp} et utilisé dans le {\ttfamily main.\+cpp} du client pour initialiser le jeu, créer des entités via des factories, et boucler sur les updates des systèmes.\hypertarget{index_autotoc_md2}{}\doxysubsubsection{\texorpdfstring{Relations clés \+:}{Relations clés :}}\label{index_autotoc_md2}

\begin{DoxyItemize}
\item L\textquotesingle{}\doxylink{classECS}{ECS} est le "{}coeur"{} \+: il stocke les entités, composants et systèmes.
\item Les factories ({\ttfamily Entity\+Factory.\+hpp}, {\ttfamily Stage\+Factory.\+hpp}) créent des entités pré-\/configurées avec des composants.
\item Les systèmes héritent de {\ttfamily \doxylink{classISystem}{ISystem}} et sont ajoutés à l\textquotesingle{}\doxylink{classECS}{ECS} pour être exécutés dans la boucle de jeu.
\item Dans {\ttfamily main.\+cpp}, tout est assemblé \+: création de l\textquotesingle{}\doxylink{classECS}{ECS}, ajout de systèmes, initialisation d\textquotesingle{}entités, et boucle update/render.
\end{DoxyItemize}\hypertarget{index_autotoc_md3}{}\doxysubsection{\texorpdfstring{2. L\textquotesingle{}\+Entity Component System (\+ECS)}{2. L'Entity Component System (ECS)}}\label{index_autotoc_md3}
L\textquotesingle{}implémentation de l\textquotesingle{}\doxylink{classECS}{ECS} est dans {\ttfamily ecs.\+hpp}. C\textquotesingle{}est une implémentation légère, optimisée pour un maximum de 2048 entités ({\ttfamily MAX\+\_\+\+ENTITIES}) et 128 types de composants ({\ttfamily MAX\+\_\+\+COMPONENT}).\hypertarget{index_autotoc_md4}{}\doxysubsubsection{\texorpdfstring{Concepts Clés de l\textquotesingle{}\+ECS \+:}{Concepts Clés de l'ECS :}}\label{index_autotoc_md4}

\begin{DoxyItemize}
\item {\bfseries{Entity}} \+: Un ID unique ({\ttfamily std\+::uint32\+\_\+t}). Les entités sont créées via {\ttfamily create\+Entity()} et détruites via {\ttfamily kill\+Entity()}. Une queue recycle les IDs des entités détruites pour éviter la fragmentation.
\item {\bfseries{Signature}} \+: Un bitset ({\ttfamily std\+::bitset\texorpdfstring{$<$}{<}MAX\+\_\+\+COMPONENT\texorpdfstring{$>$}{>}}) qui indique quels composants une entité possède. Cela permet des requêtes rapides comme {\ttfamily get\+Entities\+By\+Components\texorpdfstring{$<$}{<}T1, T2...\texorpdfstring{$>$}{>}()}.
\item {\bfseries{Component\+Type\+ID}} \+: Un ID unique pour chaque type de composant, généré via {\ttfamily get\+Component\+ID\texorpdfstring{$<$}{<}T\texorpdfstring{$>$}{>}()} (utilise un compteur statique).
\item {\bfseries{Storage\+Component$<$\+T$>$}} \+: Une classe template qui stocke les composants d\textquotesingle{}un type donné pour toutes les entités (vecteur de taille {\ttfamily MAX\+\_\+\+ENTITIES}). Elle utilise un bitset pour tracker quelles entités ont ce composant.
\item {\bfseries{\doxylink{classECS}{ECS} Class}} \+: La classe principale qui gère tout \+:
\begin{DoxyItemize}
\item Stocke les entités vivantes et une queue pour recycler les IDs.
\item Un vecteur de pointeurs vers des storages de composants ({\ttfamily std\+::vector\texorpdfstring{$<$}{<}IComponent\+Storage\texorpdfstring{$\ast$}{*}\texorpdfstring{$>$}{>}}).
\item Un vecteur de systèmes ({\ttfamily std\+::vector\texorpdfstring{$<$}{<}std\+::unique\+\_\+ptr\texorpdfstring{$<$}{<}\doxylink{classISystem}{ISystem}\texorpdfstring{$>$}{>}\texorpdfstring{$>$}{>}}).
\item Méthodes clés \+:
\begin{DoxyItemize}
\item {\ttfamily create\+Entity()} \+: Alloue un nouvel ID.
\item {\ttfamily add\+Component\texorpdfstring{$<$}{<}T\texorpdfstring{$>$}{>}(Entity e, T component)} \+: Ajoute un composant à une entité, met à jour la signature.
\item {\ttfamily get\+Component\texorpdfstring{$<$}{<}T\texorpdfstring{$>$}{>}(Entity e)} \+: Récupère un composant (pointeur).
\item {\ttfamily get\+Entities\+By\+Components\texorpdfstring{$<$}{<}T1, T2...\texorpdfstring{$>$}{>}()} \+: Retourne les entités ayant tous les composants spécifiés (via signatures).
\item {\ttfamily add\+System(std\+::unique\+\_\+ptr\texorpdfstring{$<$}{<}\doxylink{classISystem}{ISystem}\texorpdfstring{$>$}{>} system)} \+: Ajoute un système.
\item {\ttfamily update(double dt)} \+: Appelle {\ttfamily update(dt)} sur tous les systèmes.
\end{DoxyItemize}
\end{DoxyItemize}
\end{DoxyItemize}\hypertarget{index_autotoc_md5}{}\doxysubsubsection{\texorpdfstring{Relations dans l\textquotesingle{}\+ECS \+:}{Relations dans l'ECS :}}\label{index_autotoc_md5}

\begin{DoxyItemize}
\item {\bfseries{Entité → Composants}} \+: Une entité n\textquotesingle{}a pas de données intrinsèques ; elle référence des composants via son ID et sa signature.
\item {\bfseries{Composants → Storage}} \+: Chaque type de composant a son propre storage (vector$<$\+T$>$), indexé par l\textquotesingle{}ID d\textquotesingle{}entité.
\item {\bfseries{Systèmes → \doxylink{classECS}{ECS}}} \+: Les systèmes ont une référence à l\textquotesingle{}\doxylink{classECS}{ECS} ({\ttfamily \&\+\_\+ecs}) pour accéder aux entités et composants via des queries comme {\ttfamily get\+Entities\+By\+Components()}.
\item {\bfseries{Gestion de la Mémoire}} \+: Les storages sont dynamiquement alloués quand un nouveau type de composant est ajouté. La destruction des entités met à jour les signatures mais ne libère pas immédiatement les composants (confiance en la signature).
\end{DoxyItemize}\hypertarget{index_autotoc_md6}{}\doxysubsubsection{\texorpdfstring{Exemple de flux \+:}{Exemple de flux :}}\label{index_autotoc_md6}

\begin{DoxyItemize}
\item Créer une entité \+: {\ttfamily Entity e = ecs.\+create\+Entity();}
\item Ajouter un composant \+: {\ttfamily ecs.\+add\+Component(e, Position\{0.\+f, 0.\+f\});} → Met à jour la signature de l\textquotesingle{}entité et stocke dans {\ttfamily \doxylink{classStorageComponent}{Storage\+Component}\texorpdfstring{$<$}{<}Position\texorpdfstring{$>$}{>}}.
\item Query \+: {\ttfamily auto entities = ecs.\+get\+Entities\+By\+Components\texorpdfstring{$<$}{<}Position\texorpdfstring{$>$}{>}();} → Itère sur les signatures pour trouver les matches.
\end{DoxyItemize}\hypertarget{index_autotoc_md7}{}\doxysubsection{\texorpdfstring{3. Les Composants}{3. Les Composants}}\label{index_autotoc_md7}
Les composants sont des structs simples définis dans {\ttfamily \doxylink{Components_8hpp}{Components.\+hpp}}. Ils représentent des données pures, sans logique.\hypertarget{index_autotoc_md8}{}\doxysubsubsection{\texorpdfstring{Exemples principaux \+:}{Exemples principaux :}}\label{index_autotoc_md8}

\item Les systèmes itèrent sur des combinaisons spécifiques de composants (ex. \+: {\ttfamily \doxylink{classMovementSystem}{Movement\+System}} sur {\ttfamily \texorpdfstring{$<$}{<}Position, Velocity, Movement\+Pattern\texorpdfstring{$>$}{>}}).
\end{DoxyItemize}\hypertarget{index_autotoc_md10}{}\doxysubsection{\texorpdfstring{4. Les Systèmes}{4. Les Systèmes}}\label{index_autotoc_md10}
Les systèmes héritent de {\ttfamily \doxylink{classISystem}{ISystem}} (dans {\ttfamily ecs.\+hpp}), qui définit {\ttfamily virtual void update(double dt) = 0;}. Chaque système accède à l\textquotesingle{}\doxylink{classECS}{ECS} pour query les entités pertinentes et updater leurs composants.\hypertarget{index_autotoc_md11}{}\doxysubsubsection{\texorpdfstring{Liste des systèmes (dans $<$tt$>$systems/$<$/tt$>$ ) \+:}{Liste des systèmes (dans <tt>systems/</tt> ) :}}\label{index_autotoc_md11}

\begin{DoxyItemize}
\item {\bfseries{\doxylink{classInputSystem}{Input\+System}}} \+: Gère les inputs clavier/manette. Mappe des actions (Move\+Up, Shoot, etc.) et applique aux entités avec {\ttfamily \texorpdfstring{$<$}{<}Player\+Controller, Velocity, Shootable\texorpdfstring{$>$}{>}}. Relations \+: Lit SFML events, met à jour Velocity et is\+Shooting.
\item {\bfseries{\doxylink{classMovementSystem}{Movement\+System}}} \+: Met à jour {\ttfamily \texorpdfstring{$<$}{<}Position\texorpdfstring{$>$}{>}} basé sur {\ttfamily \texorpdfstring{$<$}{<}Velocity\texorpdfstring{$>$}{>}} et {\ttfamily \texorpdfstring{$<$}{<}Movement\+Pattern\texorpdfstring{$>$}{>}} (pour patterns comme Sinus, Zigzag). Relations \+: Applique dt à la position, avec modulations pour patterns ennemis.
\item {\bfseries{\doxylink{classCollisionSystem}{Collision\+System}}} \+: Détecte et résout collisions AABB sur {\ttfamily \texorpdfstring{$<$}{<}Position, Collider\texorpdfstring{$>$}{>}}. Gère dommages via {\ttfamily \texorpdfstring{$<$}{<}Health, Projectile\texorpdfstring{$>$}{>}}. Relations \+: Itère sur paires d\textquotesingle{}entités, résout overlaps, applique dégâts, détruit entités ({\ttfamily ecs.\+kill\+Entity}).
\item {\bfseries{\doxylink{classMissileSystem}{Missile\+System}}} \+: Gère les tirs via {\ttfamily \texorpdfstring{$<$}{<}Shootable, Position\texorpdfstring{$>$}{>}}. Crée projectiles (via {\ttfamily Entity\+Factory}), gère cooldowns et triple\+Shot. Relations \+: Utilise factories pour spawn, supprime hors-\/écran.
\item {\bfseries{\doxylink{classPowerUpSystem}{Power\+Up\+System}}} \+: Gère collisions power-\/ups avec joueurs, active effets comme triple\+Shot sur {\ttfamily \texorpdfstring{$<$}{<}Shootable\texorpdfstring{$>$}{>}}. Relations \+: Détruit power-\/up après collision.
\item {\bfseries{\doxylink{classWaveSystem}{Wave\+System}}} \+: Spawn vagues d\textquotesingle{}ennemis basé sur Wave\+Data (delay, count, etc.). Crée ennemis via {\ttfamily \doxylink{classECS_ab25f63dc0523958d986069505b703c12}{ECS.\+add\+Component}}. Relations \+: Timer-\/based, utilise factories implicitement.
\item {\bfseries{\doxylink{classRenderSystem}{Render\+System}}} \+: Dessine tout via SFML sur {\ttfamily \texorpdfstring{$<$}{<}Position, Drawable, Star, etc.\texorpdfstring{$>$}{>}}. Tri par layers. Relations \+: Accède à {\ttfamily \doxylink{classResourceManager}{Resource\+Manager}} pour textures, clear/draw/display sur window.
\end{DoxyItemize}\hypertarget{index_autotoc_md12}{}\doxysubsubsection{\texorpdfstring{Autres \+:}{Autres :}}\label{index_autotoc_md12}

\item {\bfseries{Flux Typique}} \+: Main → \doxylink{classECS}{ECS} (add systems/factories) → Boucle (update dt) → Systèmes (query entités/composants via \doxylink{classECS}{ECS}) → Modif composants → Render.
\item {\bfseries{Dépendances}} \+: Systèmes dépendent de composants (queries). Factories configurent entités avec composants spécifiques pour matcher les queries des systèmes.
\item {\bfseries{Avantages \doxylink{classECS}{ECS}}} \+: Scalable, facile d\textquotesingle{}ajouter systèmes/composants sans toucher le core.
\item {\bfseries{Points d\textquotesingle{}Amélioration}} \+: Ordre des systèmes non garanti (vecteur), pas de multithreading, confiance en signatures (risque de corruption).
